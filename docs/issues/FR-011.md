# FR-011: Wallet and Paystack Top-up

Status: MUS  
Labels: wallet, payments

## Requirement

Wallet credits are deducted for clearance requests, can be topped up through Paystack, and can be adjusted by admin.

## User Story

As a school owner, I want to pay for checks through wallet credits, so that pricing stays simple.

## Proposed Solution

Manage wallet credits, debits, Paystack verification, and manual adjustments.

## Implementation Flow

Use kobo integers, idempotent references, transactional wallet operations.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [x] Wallet balance is visible.
- [x] Clearance debit appears in history.
- [ ] Paystack top-up initializes and verifies server-side.
- [ ] Successful payment credits wallet once only.

## Notes

The wallet page preserves safe demo wording: client-side callbacks do **not** credit any wallet. Server placeholder endpoints now exist at `/api/wallet/debit`, `/api/paystack/initialize`, and `/api/paystack/verify`, and each intentionally returns `501` until real transactional and idempotent payment handling is added.
