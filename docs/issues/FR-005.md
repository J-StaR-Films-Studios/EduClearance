# FR-005: Admin School Approval

Status: MUS  
Labels: admin, onboarding

## Requirement

Admin can approve, reject, suspend, and edit school profiles and school users.

## User Story

As an admin, I want onboarding control, so that fake or abusive schools cannot join unchecked.

## Proposed Solution

Give platform admin control over school/user activation and suspension.

## Implementation Flow

Approve, reject, suspend, reactivate, edit official contacts.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [x] Admin can approve pending claims.
- [x] Admin can reject with note.
- [x] Admin can suspend/reactivate schools.
- [ ] Lifecycle actions are audit logged.

Mockup-driven admin school approvals, contact editing, and lifecycle controls are now implemented in the demo admin workspace. Lifecycle buttons currently surface explicit audit-log handoff notices, but persistent server-side audit writes still need to be connected before this item can be closed.
