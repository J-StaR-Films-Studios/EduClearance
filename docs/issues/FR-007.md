# FR-007: Safe Matching Results

Status: MUS  
Labels: clearance, safety

## Requirement

Clearance request matching returns no-match, possible-match, or confirmed-match wording with safe disclaimers.

## User Story

As a school user, I want careful result language, so that I do not mistake “no record” for “cleared.”

## Proposed Solution

Return careful no-match/possible-match/confirmed-match outcomes.

## Implementation Flow

Normalize search fields and match against unresolved issues.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [ ] Results support no_match, possible_match, confirmed_match.
- [x] No-match wording never says cleared.
- [ ] Possible matches are labelled as possible.
- [x] Confirmed issues show minimum useful details and actions.
