# FR-004: Authentication and RBAC

Status: MUS  
Labels: auth, security

## Requirement

Authentication and role-based authorization protect platform admin and school-only areas.

## User Story

As a platform operator, I want secure role-based access, so that only approved users see sensitive data.

## Proposed Solution

Protect platform and school areas with server-enforced roles.

## Implementation Flow

Login/logout, protected layouts, route/server-action guards.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [ ] Users can sign in/out.
- [ ] Protected routes reject unauthenticated users.
- [ ] Admin routes require platform_admin.
- [ ] School routes require active school membership.

## Notes

This build adds `/login` and `/register` as noindex placeholder entry points and preserves noindex behavior for `/claim-school`. Demo school and admin sign-in now set server-readable cookies used to enforce private-route access server-side for `/dashboard`, `/clearance/**`, `/issues/**`, `/wallet`, and `/admin/**`. Production authentication, sign-out handling, active-school membership checks, and suspension/approval enforcement still remain follow-up work.
