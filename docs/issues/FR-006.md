# FR-006: Paid Clearance Request

Status: MUS  
Labels: clearance, wallet

## Requirement

A school can start a paid clearance request for one incoming student.

## User Story

As an admitting school, I want to open a transfer clearance window, so that I can verify before admitting.

## Proposed Solution

Create the main paid workflow for one incoming student.

## Implementation Flow

Validate request, verify wallet balance, debit wallet, create case.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [x] Active school can submit clearance request.
- [ ] Request charges ₦100 from wallet.
- [x] Insufficient balance blocks request.
- [ ] Debit and request creation are transactional.

## Notes

The current school UI still simulates result states and the ₦100 charge. A server placeholder endpoint now exists at `/api/clearance/start` to document the required production contract, but it intentionally returns `501` until transactional request creation and wallet debit are implemented.
