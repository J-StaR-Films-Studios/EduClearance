# FR-010: Case Status Workflow

Status: MUS  
Labels: workflow, disputes

## Requirement

Clearance requests and issues support statuses for pending, notified, cleared, unresolved, disputed, no response, and closed.

## User Story

As all parties, I want clear statuses, so that each case has an understandable lifecycle.

## Proposed Solution

Support clear lifecycle states for requests and issues.

## Implementation Flow

Implement status transitions for cleared, unresolved, disputed, no response, closed.

## Technical Approach

Follow the stack and data model in `docs/Project_Requirements.md` and the rules in `docs/Coding_Guidelines.md`. Use server-side validation, school-scoped permissions, audit logging where sensitive, and transactional database operations for wallet/payment work.

## Key Considerations

Keep the MVP focused on clearance requests, not full school management. Use safe product language, minimize child/parent data, and avoid presenting missing records as clearance.

## Acceptance Criteria

- [ ] Statuses match the PRD enumerations.
- [x] Allowed actions update statuses predictably.
- [x] Dispute can be opened.
- [ ] Status changes are audit logged.
