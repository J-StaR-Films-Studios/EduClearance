Yes. The conversation has a **very clear MVP hiding inside it**, and my strong opinion is: **do not start with the full school-management app. Start with the “student debt/checking network” only.**

That is the sharper product. It is easier to explain, faster to build, and gives school owners a direct reason to care.

## Final product idea

Build a platform where schools in a local area can check whether an incoming student has unpaid fees or unresolved obligations from a previous school before admitting them.

The pitch is:

> “Before you admit a new student, check if they left another school owing money. Protect your school, protect other schools, and reduce bad debt across the area.”

That is much stronger than:

> “Use our school software to manage records.”

Because every school owner already understands the pain of parents leaving with unpaid balances.

## The MVP should be called something like

**EduVerify**
**SchoolCheck**
**StudentClearance**
**ClearPass Schools**
**DebtorCheck for Schools**
**EduClearance**

My favorite for now: **EduClearance**.

It sounds official, school-friendly, and not too aggressive.

## The business model

Keep it extremely simple:

### School wallet system

Each school pays **₦5,000 initial credit**.

That credit gives them **50 checks**.

So each check costs:

**₦100 per student verification.**

No revenue sharing yet. No complex splitting. No “school A gets cut, school B gets cut, platform gets cut.” That is too much logic for MVP and too hard to explain.

The first version should be:

> “Load ₦5,000. Use it to check 50 incoming students. When it finishes, top up again.”

That is clean.

Later, you can add:

> “Schools that report verified debt cases may receive reward credits.”

But not now.

## The MVP feature list

Build only these:

### 1. School directory

Before launch, you manually add all schools in the target area, for example:

**Nyanya Ward / local school cluster**

Each school profile has:

School name
Address/area
Phone number
Proprietor/head contact
Status: unclaimed / active
Date added

This is smart because even before they register, the platform already feels like a network.

### 2. School registration / claim school

A school can claim its profile.

They provide:

School name
Admin name
Phone number
Email
Password
Optional logo

You manually approve them at first. Do not overbuild verification yet.

### 3. Student debt report

A school can add a student who left while owing.

Fields:

Student full name
Gender
Previous class
Parent/guardian name
Parent phone number
Amount owed
Reason/type: school fees, books, uniform, transport, other
Academic session/term
Status: unresolved / resolved
Optional note
Date reported

Important: do **not** make it look like a public shame list. Make it look like an **internal clearance record**.

### 4. Student check

An active school searches:

Student name
Parent phone number
Previous school, if known

The system returns:

No match found
Possible match found
Confirmed record found

For MVP, fuzzy matching can be basic. Name + parent phone number is enough.

### 5. Wallet / credit deduction

Each check deducts **₦100**.

Show:

Current balance
Checks remaining
Check history
Top-up manually for now

For MVP, you do not need Paystack automation immediately. You can let schools transfer manually, then you update their wallet from the admin panel.

### 6. Admin dashboard

You need your own admin panel for:

Approving schools
Adding schools manually
Crediting wallets
Viewing searches
Viewing reports
Marking suspicious/duplicate reports
Resolving disputes

This is very important because the system can be abused if a school falsely reports a parent/student.

## The sensitive part: privacy and consent

Because this involves children, parents, school records, and financial information, you need to treat it carefully. Nigeria has the **Nigeria Data Protection Act 2023**, and the Nigeria Data Protection Commission says it is responsible for safeguarding data privacy and enforcing responsible data handling in Nigeria. ([Nigeria Data Protection Commission][1])

The NDPC also lists data-subject rights such as the right to be informed, access, rectification, objection, restriction, portability, and deletion. ([Nigeria Data Protection Commission][2])

So your MVP should include these rules from day one:

Parents must know the school may verify previous-school clearance as part of admission.
Schools must agree not to upload false, insulting, or unnecessary information.
Only verified schools can search.
Search results should show minimum necessary information.
There must be a “dispute this record” button.
Records should not be public or indexed by Google.
You need audit logs showing who searched what and when.

The safest wording is not “debtor database.” Use:

**Student Clearance Verification**
**Outstanding Balance Check**
**Transfer Clearance Record**

That sounds more professional and less legally messy.

## The MVP user flow

### For a school joining

1. School sees its name already listed.
2. Clicks “Claim School.”
3. Registers admin account.
4. You approve them.
5. They load ₦5,000 credit.
6. They can now search incoming students.
7. They can also report unresolved balances.

### For checking a new student

1. Parent comes to register child.
2. School asks: “Previous school?”
3. School searches student name and parent phone.
4. If no record: continue admission.
5. If record appears: request clearance/payment proof.
6. School can contact previous school.

### For reporting a student

1. School adds student record.
2. System marks it as “unresolved.”
3. If parent later pays, school marks it “resolved.”
4. Resolved records can still show history, but not block admission.

## What not to build yet

Do **not** build these in the first MVP:

Full result management
AI learning recommendations
Payment gateway split settlements
School fees collection
SMS receipts
Full parent portal
Attendance
Report cards
Complex multi-branch hierarchy
Automated inter-school penalties

Those are good later, but they will slow you down now.

The wedge is **clearance verification**.

## Should the app store all students?

My answer: **not fully, at least not in the MVP.**

Do not build a full “all children database.” That is heavier legally and operationally, especially because children’s data deserves extra protection. The Nigeria Data Protection Commission says it was established under the Nigeria Data Protection Act 2023 to safeguard data privacy and enforce responsible data handling in Nigeria. ([Nigeria Data Protection Commission][1]) A 2025 legal summary notes that under the NDPA, a child is anyone under 18 for personal-data processing, and controllers need appropriate mechanisms around age and consent. ([ICLG][2])

So for MVP, I would store **clearance cases**, not every student.

Meaning:

```txt
A student only enters the system when:
- a school starts a transfer clearance check, or
- a school reports an unresolved balance/clearance issue.
```

This keeps the data limited and purposeful.

Later, if the app grows, you can add a proper “student transfer profile,” but do not start there.

## Now, should adding a record cost credits?

My strong opinion:

**No, adding a record should not cost normal credits.**

Because the value of the platform depends on schools contributing records. If you charge schools to add records, many will avoid reporting, and the database becomes weak.

## Recommended credit model

### 1. Starting a clearance check costs credits

School B pays because they are receiving value.

Example:

```txt
₦100 per clearance check
```

Or better:

```txt
₦150 per clearance window
```

Because now it includes search, notification, case tracking, and contact details.

### 2. Reporting an issue is free

School A should not pay to report that a child left owing.

Reason:

The platform needs reports to become useful.

### 3. False/abusive reports cost reputation or penalty

Instead of charging upfront, punish bad behavior.

For example:

```txt
If a school repeatedly creates false reports:
- reports require admin review
- school is suspended
- school loses verification trust level
- school may pay a review fee to reactivate reporting
```

Later, you can introduce:

But not for MVP.
